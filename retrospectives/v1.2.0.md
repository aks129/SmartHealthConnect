# Retrospective — v1.2.0

**Release**: v1.2.0
**Date**: 2026-04-15
**Scope**: Wearable signals flow into skills via HealthClaw v1.3.0's Open Wearables adapter. No SmartHealthConnect code changes — SKILL.md docs only.
**Previous retrospective**: [v1.1.0](v1.1.0.md)

---

## 1. What shipped

- **Features**:
  - `healthy-habits`, `diet-exercise`, `medication-refills` SKILL.md each gain a "Wearable signals" section listing the 13 LOINC codes (heart_rate 8867-4, resting HR 40443-4, HRV 80404-7, SpO2 59408-5, respiratory rate 9279-1, body temp 8310-5, steps 55423-8, sleep duration 93832-4, VO2max 65757-1, body weight 29463-7, systolic BP 8480-6, diastolic BP 8462-4, blood glucose 15074-8) that wearable Observations now carry.
  - Skills are instructed to call `get_compiled_truth` on wearable-sourced Observations so responses can cite device Provenance (e.g. "Garmin recorded a resting HR of 58 — 14-day trend down from 62").
  - README + CLAUDE.md pick up a one-paragraph "v1.2.0 — wearable signals" note pointing at the engine.
- **Fixes**: None.
- **Breaking**: None. No new MCP tools, no client changes, no schema changes.

## 2. Engine/surface contract check

- **New FHIR reads?** No new reads in this repo — wearable Observations arrive via HealthClaw's ingestion; skills fetch them through the existing `fhir_search` they already used for non-wearable Observations. The rule that resource-specific narration goes through `get_compiled_truth` is reinforced in each updated SKILL.md.
- **New writes?** None in this repo. All write paths (including wearable ingestion) live in HealthClaw v1.3.0 behind the `wearable-sync` service-account actor declared in `action_policy.yaml` (medium risk, step_up, no human_review).
- **New PHI fields?** None. Wearable samples are redacted by HealthClaw's existing Observation redaction profile on read.
- **Skill updates?** `healthy-habits`, `diet-exercise`, `medication-refills` updated. `care-completion`, `kids-health`, `research-monitor` unchanged — wearable data is not relevant to those skills. Each updated SKILL.md still states the `get_compiled_truth` requirement inherited from v1.1.0.

## 3. CI/CD health

- No CI-relevant code changes this release (SKILL.md + docs only). No lockfile regeneration. No new tests needed — existing `fhir_search`-based code paths are already covered; the new data source is transparent.
- The engine side (HealthClaw v1.3.0) ships 29 new Python tests for the mapper, model, and routes; all 295 Python + 49 Node tests pass there.

## 4. What went well

- **Engine/surface split held under pressure.** The wearables feature is meaningful — 8 providers, 13 LOINC codes, full OAuth flow — and the SmartHealthConnect side is a zero-code change. That's the architecture working as designed.
- **LOINC/UCUM table in one place.** The code map lives in `r6/wearables/mapper.py` on the engine side; skills reference codes by number. No duplication, no drift risk.

## 5. What went poorly

- **SKILL.md is accumulating sections.** Each release has added a "Reading compiled truth" (v1.1.0) and now a "Wearable signals" section (v1.2.0). Both are valuable; three releases in, SKILL.md files may become a scroll-fest. Not acting on this yet, but noting for a future refactor.
- **No Withings support.** Open Wearables doesn't cover Withings and BP + weight scales are common. Patients running SmartHealthConnect won't get those signals until a native-API follow-up lands.

## 6. Root causes

No incidents. No regressions.

## 7. Process observations

- Commit hygiene: one logical commit for this release in each repo — appropriate for minor, docs-only scope here.
- Documentation drift: CLAUDE.md + README caught up in the same commit as the SKILL.md changes. No drift.
- Stashed WIP on `api/index.ts` + `server/routes.ts` remains — that is the user's ongoing work, deliberately not touched.

## 8. Lessons → memory

None new. Existing memories (engine/surface rule, commit hygiene, route-through-engine) all held.
